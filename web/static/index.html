Access transaction data from multiple cryptocurrencies in uniform format

Bitcoin, Ethereum (including ERC-20), Litecoin, Dogecoin, Cardano, Aptos and more coming!

Try it! See documentation

Reading blockchain data is difficult: different APIs, different responses, different transfer operations. Stocra is built with a single purpose - to simplify access to transaction data across multiple blockchains. We give you a single, uniform API to gather all transfers regardless of the underlying blockchain data structures.

Supported blockchains

The following blockchains and their cryptocurrencies are supported by stocra at this moment:
  • Bitcoin - BTC: bitcoin.stocra.com
  • Ethereum - ETH, ERC-20 token transfers: ethereum.stocra.com. Token transfers return contract address in currency_symbol field. You can get token name, symbol and scaling from /tokens endpoint
  • Litecoin - LTC: litecoin.stocra.com
  • Dogecoin - DOGE: dogecoin.stocra.com
  • Cardano - ADA: cardano.stocra.com
  • Aptos - APT, 0x1::aptos_account::transfer, 0x1::coin::transfer: aptos.stocra.com
We are constantly working on adding additional ones. These are in process of being added to our API:
  • EOS

API

We have two data structures, blocks and transactions. Blocks have height, hash, timestamp and list of transaction hashes. Transactions have hash, inputs, outputs and fees. That's it.

You can head straight to the documentation or play with our python SDK to get a feel for the API.

Transactions

We follow a simple format you might be familiar from bitcoin:
  • transaction has a set of inputs, outputs and fee
  • inputs contain source address, amount and optionally UTXO pointer
    • amount is the exact amount that was subtracted from the source address, tx fee included
    • ERC-20 transactions contain two inputs, one for the native asset to pay fees and another one for the amount of token transferred (without scaling)
    • failed ERC-20 transactions have only fee input
  • outputs contain destination address and amount
Addresses in both inputs and outputs are in the same format as returned from node, no address conversion is done.

Versioning

Our API follows SEMVER versioning (without patch versions) and version is specified in the URL. Response formats are guaranteed to not change within the MAJOR.MINOR version. New MINOR versions will be backward compatible but might contain new fields. New MAJOR versions have no guarantees about compatibility but we will give you a heads-up and enough time to upgrade. Deprecated versions will have deprecation header for at-least a month before they are shut down. New versions and deprecations will also be announced through our newsletter.

Limits

The default limit for unauthenticated user is 10,000 requests per blockchain per day. Once your limit is depleted you will receive responses with status code 429. The response will also contain Retry-After header with the amount of seconds remaining until your limit is reset.

Do you need more than that? Do you prefer dedicated deployment, perhaps even in the same cloud as your app? Reach out to us.
